Implementation plan · July 2026

Never generate the same reasoning twice.

Make the rendered chat timeline the only authority for “what came before this turn,” capture that fact before mutating the timeline, and keep the reasoning decision inside the generation workflow.

✓
The rule can be absolute.

If the last eligible, persisted text message in the chat view is reasoning, reuse it and generate only the assistant response. Otherwise, generate reasoning when the setting is enabled.

Diagnosis

Why the current check still fails

Wrong source
useChatGeneration.ts asks LLMChatProjection.GetMessages(). That projection describes model context, not the chat view. Retention, chapter buffers, and prompt assembly can make those two histories differ.
Empty retry
The lookup only runs when userInput.trim() is non-empty. An empty continuation after response generation fails therefore hard-codes skipReasoning = false and creates the duplicate.
Leaky policy
The React hook decides whether reasoning runs, then passes a boolean into TextGenerationService. Any other caller can accidentally bypass the same policy.
Invariant

One selector, one meaning

Add a named selector to the user-facing projection. Its contract must match the visible timeline, not the request sent to a model.

Generate reasoning iff reasoning is enabled and the turn-start message is not a reasoning message.
  • ✓ Include persisted user-message, system-message, assistant, and reasoning entries.
  • × Exclude deleted, hidden, chapter-covered, or book-covered entries exactly as the chat view does.
  • × Exclude transient streaming/regeneration placeholders; only event-backed messages can anchor a new turn.
  • × Exclude story prompts, response prompts, memories, notes, plans, images, and other non-conversation artifacts.

Ownership

Where the fix belongs

  1. Define the timeline selector

    Add GetLastPersistedTextMessage() to UserChatProjection.ts. Keep the eligibility predicate beside GetMessages() so the render filter and anchor filter cannot drift silently.

  2. Make one service own the turn boundary

    Change the send workflow so TextGenerationService receives the user input, captures the anchor, persists the input, and applies the reasoning rule in that exact order.

  3. Delete the hook heuristic

    Remove the hasUserInput, contextMessages, and skipReasoning calculation from useChatGeneration.ts. The hook should initiate a turn, not interpret history.

  4. Keep regeneration separate

    regenerateResponse(messageId) continues to regenerate only the selected assistant response. It must never enter the new-turn reasoning path.

Guardrail

No content sniffing

Never infer reasoning from [Reasoning] markers or prompt text. Message type and persisted timeline position are the only inputs.

Guardrail

No LLM projection

Model context is allowed to compress, retain, reorder, or append instructions. Those transformations must not affect UI turn semantics.

Guardrail

No timing guess

Capture the anchor synchronously before the first awaited mutation. Persist the anchor ID in the turn object for diagnostics and tests.

Verification

Tests that make the invariant difficult to regress

  • Selector returns reasoning when later assistant entries are deleted or hidden.
  • Selector ignores streaming and regeneration placeholders.
  • Empty continuation after failed response makes zero reasoning API calls.
  • Non-empty send after a reasoning anchor makes zero reasoning API calls.
  • Request prompts appended after chat history cannot change the decision.
  • Assistant/user anchor generates exactly one reasoning message when enabled.
  • Disabled reasoning makes zero reasoning API calls for every anchor type.
  • Response regeneration edits in place and makes zero reasoning API calls.
Proposed scope: projection selector, send-turn ownership, focused unit tests, and removal of the existing hook heuristic. No event migration or prompt-format change required.
