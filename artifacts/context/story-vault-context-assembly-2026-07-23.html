What the model actually sees

Text reaches the model in two passes: the event history is first compressed into a projected chronology, then durable context is floated near the recent conversation before a request-specific instruction is appended.

1 · Compress the past Books replace chapters; chapters replace covered chat and reasoning.
2 · Re-anchor durable facts Memories and tracked, active Character Sheets sit just before the last N projected items.
3 · Finish for the request A response prompt or reasoning prompt closes the context; optional guidance comes last.

Recency bridge

It remains while fewer than 6 visible items follow the latest chapter, then drops. The setting can be 0.

Lifecycle filters

Deleted, hidden, and expired items leave

Notes can expire; older Plan versions are hidden; reasoning has its own retention window.

Pass two · LLMMessageContextService

2Float durable context toward recency

The projected chronology is split before its last T items. Durable context is inserted into that split, keeping it near the model’s attention while preserving the newest turns after it.

[ 1 … length − T ] Earlier projected chronology Story, summaries, messages, Notes, Plans, reasoning—whatever survived pass one.
+
system message # Memories All non-empty memory content joined in stored order.
+
system message # Character Sheets Tracked, active characters with non-empty sheet bullets.
+
last T projected items Recent projected chronology T defaults to 5 and is configurable from 0–50 per chat.
!
Current code order: Memories, then Character Sheets. Both are inserted before the final T projected items, not after a special “messages only” count. In a very short chat, they can even precede # Story in the final request although # Story is first inside the projection.

Memories are broad and durable

  • Every non-blank memory is included.
  • There is no active/inactive filter in this builder.
  • Content is concatenated into one system message.

Character Sheets are selective

  • Only tracked and effectively active characters are included.
  • Only non-empty sheetItems become bullets.
  • The appearance field does not enter text-generation context here.

Source-by-source reference

What survives, where it lands, and why

“Chronological” below means chronological inside the projected history, before the durable-context split moves Memories and Character Sheets near the end.

Book If visible system · marked summary Inserted at the first covered chapter’s position and hides all covered chapters.
Chapter If not book-covered system · marked summary Hides covered message/reasoning items. The newest chapter may temporarily receive its trailing buffer.
Trailing chapter items Temporary original roles Last N covered IDs are inserted before the latest chapter until N visible items follow it. Default N = 6.
Chat messages Unless hidden/deleted user / assistant / system Chronological. Covered messages are replaced by their chapter summary outside the temporary buffer.
Notes Until expiry system · [Note] Chronological and not hidden by chapters. Expiry counts later regular message events; null means permanent.
Plans Latest visible state system · named markers Chronological and not hidden by chapters. Refresh hides prior instances for that Plan definition.

Related request variants

The same ingredients are reused differently

These are not the normal two-call chat path, but they explain why inspecting a different generation workflow may show a different final context.

Regenerate a response

Projected history is truncated before the target assistant message, then Memories and Character Sheets are inserted. The chat response prompt is omitted; optional feedback is appended as a user message quoting the prior response.

Draft a chapter

A captured message snapshot receives reasoning-retention filtering, Memories, and Character Sheets. A user instruction combining chapter-summary and chapter-title prompts is appended, requesting one JSON object.

Summarize or title a book

This is a narrower path: tracked, active Character Sheets, the supplied chapter summaries, then the book summary/title prompt. Memories and ordinary chat history are not included.

Recommended cleanup · correctness first

Refactor toward one explicit context document

Preserve the event format and external behavior while separating event reduction, selection policy, durable-context composition, and request rendering.

Fix before extraction

Deleted chapters can still be “latest”

getLastChapter() ignores the deleted flag. Existing tests acknowledge that deleting a chapter can duplicate restored messages through the trailing buffer.

Fix before extraction

The trailing buffer can duplicate visible items

It rehydrates covered IDs without restricting them to chapter-hidden message/reasoning types or deduplicating against the visible list. A covered Plan can therefore appear twice.

P0
Correctness

Make chapter buffering safe

Select the latest non-deleted chapter, buffer only items actually hidden by that chapter, and enforce unique message IDs in the result.

  • Turn the acknowledged deletion behavior into a failing regression test.
  • Add invariants for stable order and no duplicate IDs.
  • Cover Plans, Notes, Clarifications, deleted items, and a book-hidden latest chapter.
P1
Clarity

Fix vocabulary and delete dead branches

Rename private “story prompt” APIs to “response prompt”; the actual Story is the # Story event. Give Story its own item type instead of representing it as a generic message.

  • Carry includeResponsePrompt consistently through private methods.
  • Merge identical book-summary and book-title assemblers.
  • Remove identical “full/simple” chapter formatting and runtime-unused helpers.
P2
Isolation

Replace mutable setters with a pure projection policy

The cached projection currently receives request-time settings through SetPreviousChapterMessageBuffer and SetReasoningRetention. Pass an immutable policy to selection instead.

  • Use selectMessages(state, policy) with no projection mutation.
  • Normalize defaults once into a typed ContextPolicy.
  • Make the counting basis explicit: visible items, regular messages, or user turns.
P3
Consolidation

Build one snapshot, then render each request

Load settings, prompts, Memories, and Character Sheets once per request. Compose a typed context document, then render it as response, reasoning, regeneration, chapter, or book messages.

  • Share one durable split instead of rebuilding it for consolidated reasoning.
  • Avoid fetching chat settings twice when resolving the reasoning prompt.
  • Fetch independent sources in parallel and freeze them for the request.
P4
Confidence

Add traces, parity fixtures, and linear-time filters

Every included item should retain its source and every excluded item should have a reason. Use those traces for Request Inspection and exact old-vs-new parity tests.

  • Golden fixtures for response, reasoning, regeneration, chapter, and book paths.
  • One reverse pass for note/reasoning age instead of repeated slices and scans.
  • Index message state by ID if stress measurements justify it.

Target shape

Facts in; policy applied once; renderers out

The useful consolidation is a shared intermediate representation—not forcing every workflow to include the same sources.
