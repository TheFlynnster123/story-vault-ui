Story Vault UI · implementation planning

Character sheets with less noise and more control.

Three separate plans refine review diffs, add a deterministic “do not track” record state, and allow trusted characters to accept AI changes automatically.

Strategy: keep the LLM contracts strict, then enforce user policy and change detection in deterministic application services before anything reaches persistence or story context.
Plan 1 · Diff-only review Match stable items regardless of index and render only additions/removals. Plan 2 · Tracking opt-out Keep a name record while excluding it from every AI-managed character path. Plan 3 · Auto-accept Apply trusted existing-character changes and retain the remainder for review.

Shared implementation seam

Character record Persisted identity, sheet, activity, and per-character policy.
Selection + sync Strict structured LLM responses remain complete and validated.
Maintenance policy Filters tracking, computes real changes, and routes acceptance.
Persistence + context Only eligible approved data reaches text or image generation.
Plan 1

Show only meaningful sheet diffs

Review cards display only added and removed facts. An unchanged fact is matched by normalized text, even when the LLM returns it at another index; reorder-only responses create no sheet change.

No schema change
Step 1

Create one domain diff helper

Compare normalized, deduplicated item text without using array indexes. Return added and removed while preserving each source list’s display order.

Step 2

Use the helper for change detection

Replace the order-sensitive areSheetItemsEqual check in CharacterMaintenanceService. Do not create sheet proposals when both diff lists are empty.

Step 3

Render additions and removals

Replace “Current sheet / Proposed sheet” in the review modal with compact “Added” and “Removed” groups. Keep activity changes visible beside the sheet diff.

Step 4

Lock behavior with tests

Cover insert, delete, replacement, empty lists, reordered matches, mixed activity, and a fact moving from index 0 to index N.

Decision · exact normalized matching

  • Trim and deduplicate through the existing normalizeSheetItems rules.
  • Match case-sensitively by value, not position; wording or capitalization edits remain visible.
  • Keep the LLM’s complete-replacement response contract for one-to-one validation and safe apply.

Verification

  • A pure reorder produces no proposal.
  • Unchanged items never appear in the modal.
  • Applying still writes the full proposed replacement list.
  • Activity-only changes still render and can be decided.
Primary files: CharacterUpdateProposal.ts, CharacterMaintenanceService.ts, CharacterUpdateReviewModal.tsx, and their focused tests.
Done when Mara’s unchanged “Navigator” fact is absent from review even if it moved, while only “Removed: Carries the map” and “Added: Carries the brass key” remain.
Plan 2

Add a deterministic tracking opt-out

A character can remain in the roster as a name-bearing record while being excluded from AI activity detection, sheet synchronization, durable text context, and image appearance injection. Manual editing remains available.

Record + pipeline
Step 1

Persist the policy

Add isTracked to CharacterDescription, default it to true, migrate missing values to true, and bump the character schema.

Step 2

Expose “Track with AI”

Add a per-character switch and an “Not AI tracked” badge. When off, disable Generate/Update Sheet and explain that the record is retained as an exclusion rule.

Step 3

Enforce at every LLM boundary

Keep all names in the known roster, but discard selections matching untracked records. Never send them to sheet sync, never change their detected activity, and reject stale proposals targeting them.

Step 4

Filter all generated context

Exclude untracked sheets in LLMMessageContextService and return no appearance context from both current and legacy image-generation lookup paths.

Deterministic guardrails

  • The name stays in knownCharacters, labeled as untracked for the prompt.
  • Response normalization checks the stored record and drops its ID/name regardless of model output.
  • The proposal apply service rechecks current tracking state, protecting against old pending proposals.
  • Turning tracking off updates updatedAt, so already-prepared changes become stale.

Verification

  • Legacy records migrate as tracked and preserve existing behavior.
  • A POV name returned by the model is neither updated nor recreated as a duplicate.
  • No POV sheet or appearance appears in text/image LLM inputs.
  • Manual name, sheet, appearance, and image-model edits still save.
Risk to handle: simply filtering the record before active-character selection would let the LLM rediscover the same name as “new.” The full roster must be retained for matching, with eligibility filtered after matching.
Primary files: CharacterDescription.ts, CharacterDescriptionsService.ts, CharacterCard.tsx, CharacterDescriptionsPage.tsx, ActiveCharacterSelectionService.ts, CharacterMaintenanceService.ts, LLMMessageContextService.ts, ImageGenerator.ts, and ImageGenerationService.ts.
Done when An untracked POV character remains visible and manually editable in Characters, but repeated automatic runs cannot select, update, inject, or recreate that character.
Plan 3

Auto-accept changes for trusted characters

Existing tracked characters can opt into automatic acceptance. Mixed batches apply opted-in changes through the same conflict-checked service and leave all other changes in the existing review flow.

Per-character policy
Step 1

Persist an opt-in flag

Add autoAcceptChanges, default false. If Plan 2 ships in the same change set, migrate both flags in one schema bump; otherwise use the next version.

Step 2

Add the character toggle

Show “Automatically accept AI changes” on each card with an opt-in badge. Disable it while AI tracking is off because an untracked record cannot receive AI changes.

Step 3

Partition prepared changes

In the maintenance policy, split changes by current stored character preference. Existing opted-in records auto-apply; new characters and opted-out records remain reviewable.

Step 4

Reuse safe apply semantics

Route the automatic subset through applyUpdateProposal so updatedAt conflicts, normalization, one-save behavior, and sheet-source metadata remain identical to manual approval.

Acceptance policy

  • Opt-in applies to both scheduled and manually requested AI sheet/activity changes.
  • New characters always require review because no persisted record owns the preference yet.
  • Tracking off takes precedence over auto-accept.
  • A mixed batch may return “auto-applied N; M ready for review” for clear feedback.

Verification

  • All opted-in changes apply with no pending proposal.
  • Mixed batches apply only opted-in IDs and persist the remainder.
  • Conflicts fall back to review; no partial conflicted write occurs.
  • One character’s toggle cannot accept another character’s change.
Risk to handle: character records and pending proposals use separate managed blobs. Centralize the partition/apply/save sequence in one service, preserve the full proposal on an apply conflict, and test failure ordering so reviewable changes are not silently lost.
Primary files: CharacterDescription.ts, CharacterDescriptionsService.ts, CharacterCard.tsx, CharacterDescriptionsPage.tsx, CharacterMaintenanceService.ts, CharacterUpdateProposalService.ts, settings/status copy, and focused tests.
Done when A batch for auto-accept Mara and review-required Ivo immediately updates Mara, leaves only Ivo in the review control, and keeps new-character proposals manual.
Grounded in the current schema-v3 character record, complete-replacement sheet sync, proposal review flow, and centralized text/image context builders. Recommended delivery order: Plan 1 first; Plans 2 and 3 together if convenient, sharing one schema migration.
