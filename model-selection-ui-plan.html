Interaction principles

Preview before committing

Clicking a setup changes the inspector. Only the primary footer action changes the application.

Return to the exact current state

Reopening highlights the active setup with its real configuration, making small edits feel continuous.

Make setups primary

Saved, current, and recent configurations appear before the raw provider catalog.

Make inheritance explicit

Say “Use system default” or “Use chat model” and show which setup will actually be inherited.

Implementation sequence

Ship value before changing persistence

Phase 01

Draft, preview, apply

Separate row selection from commitment. Add modal draft state, explicit Cancel and Apply actions, and preserve in-session edits while comparing models.

Phase 02

Two-pane picker

Extract the catalog, setup list, inspector, and request-settings form. Use list-to-detail navigation on small screens.

Phase 03

Editable setups

Promote presets into the main object. Support rename, update, duplicate, delete, and save-as-new while migrating existing data unchanged.

Phase 04

Configuration-aware history

Record recent setups instead of model IDs, then standardize the picker across settings, prompts, plans, chats, and reasoning.

Recommended first release

Build phases 1 and 2 together. They directly remove the “rebuilt every time” feeling without requiring a storage migration. Then promote presets into fully editable setups.
